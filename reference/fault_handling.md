# 故障处理

核心原则：用户可以接受"还有 15 个没翻译完，原因是 XXX"，但无法接受"全部完成"后发现一半是空的。所以我们选择如实报告，而非美化结果。

## 异常日志

执行过程中，所有异常追加写入 `ai_analysis/ERRORS.md`：

```markdown
# 执行异常日志

## [阶段标记] 异常类型
- **影响范围**: 哪些提示词/文件受影响
- **原因**: 具体错误信息
- **处理**: 采取了什么措施（重试/跳过/降级）
- **状态**: 已解决 / 未解决
```

## 故障分类与处理

| 故障类型 | 识别方式 | 处理策略 | 阻断流程？ |
|---------|---------|---------|----------|
| 子代理权限拒绝 | 返回中包含 "denied"/"permission" | 重试 2 次 → 主 agent 兜底 → 记录 | 否 |
| 子代理超时 | 长时间无返回 | 拆分为更小批次重试 → 记录 | 否 |
| 子代理静默失败 | 声称完成但 Glob 找不到文件 | 重新派发 → 记录 | 否 |
| 源文件无法读取 | Read 返回错误 | 翻译文档中标注 [源文件无法读取] → 记录 | 否 |
| 写入失败 | Write 返回错误 | 重试 1 次 → 记录 | 否 |
| MANIFEST 丢失 | Glob 找不到 MANIFEST* | 重新执行第二阶段搜索 | 否，回退重做 |
| 项目无 LLM 内容 | 第一阶段未发现提示词或依赖 | 告知用户并终止 | **是** |
| 搜索结果为空 | 8 种搜索方法全部返回空 | 确认搜索模式正确后如实告知用户 | **是** |

### 子代理失败的分级重试

**第一级：重新派发子代理（优先）**
1. 检查子代理返回状态——错误或权限拒绝时标记该批次为未完成
2. 调整 prompt 后重新派发（权限问题：预先说明需要权限或拆分批次；超时：拆分为更小单元；内容错误：补充更明确的指令）
3. 同一批次最多重试 2 次

**第二级：主 agent 兜底（仅在子代理反复失败时）**
4. 连续 2 次子代理失败后，主 agent 直接用 Write 工具补写
5. 兜底时只读提示词正文，跳过周围代码，避免消耗过多上下文

## 健康检查

每次关键操作后做轻量确认，确保操作实际生效：
- 写文件后 → Glob 确认文件存在（不需要读内容）
- 子代理返回后 → Glob 检查该批次文件数量 = 预期数量
- 阶段完成后 → 检查 ERRORS.md 有无未解决异常
- 验证通过后 → 最终文件计数 = MANIFEST 条目数

发现异常时：记录到 ERRORS.md → 尝试修复 → 修复失败标记"未解决" → 继续后续步骤但最终报告中披露。

## 降级交付

经过所有重试后仍有部分无法完成时，允许降级交付，但需要满足：

1. **所有失败都已记录在 ERRORS.md**（文件名、原因、已尝试的修复）
2. **最终报告包含"未完成项"清单**：
   ```
   ⚠ 未完成项（共 X 个）：
   1. [文件名] - 原因：[具体原因]
   2. [文件名] - 原因：[具体原因]
   建议用户操作：[如何手动补完或重新执行]
   ```
3. **INDEX.md 中对未翻译的提示词标记 [未翻译]**
4. **覆盖率如实报告**（如"93%（67/72），5 个因 XXX 未完成"）

## 诚实报告原则

这些规则的本质是"数字要精确、事实要可验证"：
- 报告数量时用精确数字（"已创建 38 个"），不用"60+"、"大部分"等模糊词
- 子代理声称创建了 N 个文件，必须用 Glob 验证磁盘上实际存在 N 个文件后才能采信
- ERRORS.md 有未解决异常时，在报告中注明并引导用户查看
- 如果中途发现之前阶段有遗漏，回头补齐再继续——不要带着已知缺口向前推进
- 如果出现无法解决的故障，立即告知用户（出了什么问题、已尝试什么、建议怎么做）
