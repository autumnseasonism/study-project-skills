# 故障处理与诚实报告

**核心原则：宁可如实报告失败，也不虚报成功。用户可以接受"还有 15 个没翻译完，原因是 XXX"，但无法接受"全部完成"后发现一半是空的。**

## 故障日志持久化

在执行过程中，将所有异常事件记录到 `ai_analysis/ERRORS.md`。该文件是追加写入的，格式：

```markdown
# 执行异常日志

## [时间戳或阶段标记] 异常类型
- **影响范围**: 哪些提示词/文件受影响
- **原因**: 具体错误信息
- **处理**: 采取了什么措施（重试/跳过/降级）
- **状态**: 已解决 / 未解决
```

## 故障分类与处理策略

| 故障类型 | 识别方式 | 处理策略 | 是否阻断流程 |
|---------|---------|---------|------------|
| **子代理权限拒绝** | 子代理返回中包含"denied"/"permission"关键词 | 重试2次 → 主agent兜底 → 记录ERRORS.md | 否，降级处理 |
| **子代理超时** | 子代理长时间无返回 | 拆分为更小批次重试 → 记录ERRORS.md | 否，降级处理 |
| **子代理静默失败** | 子代理声称完成但 Glob 检查文件不存在 | 标记为未完成，重新派发 → 记录ERRORS.md | 否，重试处理 |
| **源文件无法读取** | Read 工具返回错误 | 在翻译文档中标注 [源文件无法读取]，记录ERRORS.md | 否，标记跳过 |
| **翻译文档写入失败** | Write 工具返回错误 | 重试1次 → 记录ERRORS.md | 否，记录失败 |
| **MANIFEST 文件丢失** | 恢复时 Glob 找不到 MANIFEST* | 重新执行第二阶段搜索 | 否，回退重做 |
| **项目无 LLM 相关内容** | 第一阶段未发现任何提示词或LLM依赖 | **立即告知用户并终止**，不要硬凑内容 | **是，终止** |
| **搜索结果为空** | 8种搜索方法全部返回空 | 确认搜索模式正确后，如实告知用户 | **是，终止** |

## 每步操作后的健康检查

每次关键操作后，执行一个轻量健康检查，确认操作实际生效：

```
写文件后  → Glob 确认文件存在（不需要读内容，只确认存在）
子代理返回后 → Glob 检查该批次文件数量 = 预期数量
阶段完成后 → 读取 ERRORS.md 检查是否有未解决的异常
验证通过后 → 最终文件计数 = MANIFEST 中的条目数
```

**如果健康检查发现异常**：
1. 记录到 ERRORS.md
2. 尝试修复（重试/降级）
3. 如果修复失败，在 ERRORS.md 中标记为"未解决"
4. **继续执行后续步骤**（不要因为个别失败阻断整个流程），但在最终报告中如实披露

## 禁止的掩盖行为

以下行为**严格禁止**，因为它们会让用户误以为任务成功完成：

1. **禁止吞掉错误**：任何操作失败都必须记录到 ERRORS.md，不能静默忽略
2. **禁止用成功措辞包装失败**：如果有 5 个文件翻译失败，不能说"翻译完成"，必须说"翻译完成 67/72 个，5 个失败（详见 ERRORS.md）"
3. **禁止跳过验证步骤**：不能因为"大部分都成功了"就跳过验证，每个阶段的出口验证必须执行
4. **禁止将子代理的乐观报告当作事实**：子代理说"已创建 10 个文件"，必须 Glob 验证实际存在 10 个文件
5. **禁止在遇到困难时降低标准**：不能因为反复失败就放弃某些提示词而不告知用户

## 降级交付

如果经过所有重试和兜底措施后，仍有部分任务无法完成，**允许降级交付**，但必须满足以下条件：

1. **所有失败都已记录在 ERRORS.md 中**，包含：失败的具体文件、原因、已尝试的修复措施
2. **向用户的最终报告中包含一个醒目的"未完成项"清单**：

```
⚠ 未完成项（共 X 个）：
1. [文件名] - 原因：[具体原因]
2. [文件名] - 原因：[具体原因]
...
建议用户操作：[如何手动补完或重新执行]
```

3. **INDEX.md 中对未翻译的提示词标记 [未翻译]**，不要假装它们不存在
4. **覆盖率报告必须反映真实情况**：如"覆盖率 93%（67/72），5 个因 XXX 原因未完成"
