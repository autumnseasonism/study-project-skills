# 按规模分级的执行策略

根据第二阶段识别出的提示词数量选择策略。核心思路：规模越大，主 agent 越要放手——把内容生产下放给子代理，主 agent 只做调度和验证。

为什么要这样分级？主 agent 的上下文窗口是最宝贵的资源。如果主 agent 亲自 Read 50 个源文件的内容，上下文会被撑满，导致后半程质量骤降甚至丢失 MANIFEST 等关键信息。让子代理承担内容生产，主 agent 专注全局调度，是大型项目成功执行的关键。

## 小型（≤30 个提示词）

主 agent 可以直接操作：
- 搜索阶段可以 Read 源文件内容来确认提示词
- 翻译阶段可分 2-3 个子代理并行完成，也可以主 agent 直接写
- 验证阶段主 agent 自己比对文件列表即可
- 报告阶段可以主 agent 直接写，也可以委派子代理

## 中型（31-100 个提示词）

主 agent 做调度和验证，子代理做内容生产：
- **主 agent 不读取源文件内容**——所有源文件阅读和翻译文档写入都在子代理中完成
- 主 agent 只负责：(a) 分配任务给子代理，(b) 用 Glob 验证子代理输出的文件是否存在
- 子代理 prompt 中给出翻译模板文件路径，让子代理自己读模板、读源文件、翻译、写文件
- 按功能模块分组，每个子代理处理 5-10 个提示词，并行不超过 6 个，分波调度
- 验证阶段的映射比对可委派给一个 Explore 子代理完成
- 报告和说明书委派给独立子代理生成

## 大型（101-300 个提示词）

在中型策略基础上增加**模块级子协调器**：
- MANIFEST 按模块拆分（每个 `MANIFEST_[模块名].md` 不超过 30 条）
- 为每个模块派发一个协调器子代理，负责：
  1. 读取本模块的 MANIFEST
  2. 按 5-10 个一批拆分并逐批翻译
  3. 完成后用 Glob 自验证本模块文件齐全性
  4. 返回简短状态报告：`模块X: 已完成 M/N 个，缺失: [列表或"无"]`
- 主 agent 只需：(a) 生成各模块 MANIFEST，(b) 并行派发协调器（≤6 个），(c) 收集状态，(d) 对报告缺失的模块重新派发
- 验证阶段：派发 Explore 子代理读取所有 MANIFEST 并与磁盘文件交叉比对

## 超大型（300+ 个提示词）

在大型策略基础上：
- 协调器分波执行（每波 ≤4 个，确保稳定性）
- INDEX.md 改为按模块生成独立索引文件（`INDEX_[模块名].md`），最后由子代理合并为总索引
- 报告分章节由不同子代理并行生成，最后合并
- 主 agent 全程只做最顶层调度，不接触任何提示词内容

## 子代理调度通用规则

- 并行不超过 6 个——过多时难以追踪状态且容易遗漏失败的代理
- 每波全部返回后再派发下一波，每波之间更新 MANIFEST 翻译状态
- 每个子代理完成后用 Glob 验证文件实际写入
- 同一批次最多重试 2 次子代理，仍失败则主 agent 兜底（详见 [fault_handling.md](fault_handling.md)）
- 不得在任何子代理仍在运行时向用户报告"完成"

## 上下文节约的具体做法

- 主 agent 搜索阶段只记录文件路径列表，需要了解内容时委派 Explore 子代理返回摘要
- 子代理 prompt 中给出模板文件路径让子代理自行 Read，不内联模板内容
- 验证阶段只比对文件名列表，不读取翻译文档内容
- 向用户汇报时使用简洁的统计数字和表格
