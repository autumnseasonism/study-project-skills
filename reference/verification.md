# 验证流程

本文件包含 MANIFEST 格式定义、第三阶段出口验证和最终交付验证的详细流程。SKILL.md 中引用此文件，执行到验证步骤时读取。

## MANIFEST 格式

```markdown
# 提示词主清单

> 自动生成于第二阶段搜索完成后，用于第三阶段验证比对。

## [分类1] (X 个)
| # | 源文件路径 | 变量名/提示词名 | 简述 | 翻译状态 |
|---|-----------|---------------|------|---------|
| 1 | `path/to/file.md` | VARIABLE_NAME | 一句话描述 | 待翻译 |
| 2 | ... | ... | ... | 待翻译 |

## [分类2] (Y 个)
| # | 源文件路径 | 变量名/提示词名 | 简述 | 翻译状态 |
|---|-----------|---------------|------|---------|
| 3 | ... | ... | ... | 待翻译 |

---
总计: N 个（其中中文原文 M 个）
```

## 第三阶段出口验证（6 步）

验证的目的是确保每个提示词都有对应的翻译文档。跳过验证是交付不完整的最常见原因——用户不会逐个检查几十个文件，所以自动化验证是唯一的保障。

**步骤 1：建立源文件清单**
读取 MANIFEST 文件（`MANIFEST.md` 或多个 `MANIFEST_*.md`），获取完整的提示词列表。如果 MANIFEST 因故不存在，用 Glob/Bash 重新扫描源文件重建。

**步骤 2：建立已生成清单**
用 Glob 扫描 `ai_analysis/translated_prompts/` 下所有 `.md` 文件（排除 INDEX.md 和 MANIFEST*.md），生成"已翻译清单"。

**步骤 3：逐一比对映射**
将每个源文件与已翻译文件进行一对一映射匹配。不能仅靠文件数量比较——可能有多余文件掩盖缺失。对于大量文件，可以委派 Explore 子代理执行比对（避免主 agent 消耗大量上下文处理长列表）。输出差异表：
- 已覆盖：源文件名 → 翻译文件名
- 未覆盖：源文件名 → [缺失]
- 多余/重复：翻译文件名 → [无对应源文件或重复]

**步骤 4：补缺**
如果存在未覆盖的源文件，委派子代理补写翻译文档（遵循 fault_handling.md 中的重试策略）。补缺完成后，重新执行步骤 2 和步骤 3 确认全部覆盖。

**步骤 5：生成 INDEX.md**
只有在步骤 3 确认 0 个未覆盖时才生成 INDEX.md。INDEX.md 条目数必须与实际 `.md` 翻译文件数一致。同时更新 MANIFEST 中所有条目的翻译状态为"已完成"。

**步骤 6：向用户报告**
报告时包含：总源文件数、已翻译数、覆盖率百分比。如有重复文件需说明原因。

## 阶段间门控

每个阶段必须满足出口条件后才能进入下一阶段：

| 阶段 | 出口条件 | 验证方法 |
|------|---------|---------|
| 第一阶段 | 项目根目录已确认，LLM 依赖已识别 | 输出项目概况摘要 |
| 第二阶段 | 所有搜索方法已执行，MANIFEST 已生成 | 输出精确的提示词总数和分类计数 |
| 第三阶段 | 每个提示词都有翻译文档，INDEX.md 已生成 | 执行上述 6 步验证流程 |
| 第四阶段 | 报告文件已写入磁盘 | Glob 确认文件存在，Read 检查非空 |
| 第五阶段 | 说明书文件已写入磁盘 | Glob 确认文件存在，Read 检查非空 |

已知少量失败可以通过门控（前提是失败已记录到 ERRORS.md 且已尝试修复），但未检测的失败不能通过。区别在于"知道有 3 个没翻译完"（可以继续）vs"不知道有多少没翻译完"（不能继续）。

## 最终交付验证

所有五个阶段完成后，在向用户报告最终结果之前执行全局验证：

```
□ ai_analysis/translated_prompts/ 下文件数量 = [精确数字]
□ 源提示词文件数量 = [精确数字]
□ 覆盖率 = [百分比]（目标 100%，如有缺失列出具体文件和原因）
□ INDEX.md 存在且条目数与翻译文件数一致
□ AI_MODEL_USAGE_ANALYSIS.md 存在且非空
□ PRODUCT_GUIDE.md 存在且非空
□ ERRORS.md 状态 = [无异常 / 有 N 个未解决异常]
```

发现问题时：
1. **可修复的** → 立即修复后重新验证
2. **不可修复但已记录的** → 在最终报告中如实披露，允许降级交付
3. **不可修复且未记录的** → 先记录到 ERRORS.md，再按第 2 条处理
