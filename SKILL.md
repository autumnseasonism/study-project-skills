# Study Project

学习分析当前工作目录下的开源大模型项目，提取提示词和工具定义，生成分析报告和产品说明书。

## 为什么需要这个流程

开源 LLM 项目中，提示词散落在代码各处（变量定义、配置文件、API 调用参数），很难快速掌握全貌。这个 skill 通过系统化搜索把它们全部找出来，翻译成中文，并梳理出项目的 LLM 使用架构。最终产出三样东西：翻译文档、技术分析报告、面向普通用户的产品说明书。

## 工作流程

五个阶段依次执行，每完成一个阶段简要汇报进度。

### 第一阶段：项目结构探索

先找到项目根目录——从当前目录向上查找 `.git/`、`package.json`、`pyproject.toml`、`go.mod`、`Cargo.toml` 等标志文件。找不到则用当前目录并提示用户确认。

然后快速建立项目认知：
1. 读 `README.md` 了解项目定位
2. 读依赖文件（`requirements.txt`、`package.json` 等）识别用了哪些 LLM 库（`langchain`、`openai`、`anthropic`、`crewai`、`autogen`、`llamaindex`、`dspy`、`instructor` 等）
3. 扫描目录结构，定位 `src/`、`prompts/`、`templates/`、`config/` 等关键目录
4. **如果没有发现任何 LLM 相关依赖或提示词文件，告知用户并终止**——避免在非 LLM 项目上浪费时间
5. 评估项目规模：超过 1000 个文件则先扫核心目录（`src/`、`app/`、`lib/`），小项目全量扫描

搜索时排除依赖和构建目录：`node_modules/`、`vendor/`、`site-packages/`、`.venv/`、`venv/`、`dist/`、`build/`、`__pycache__/`。

### 第二阶段：提示词与工具识别

这是核心搜索阶段。读取 `templates/search_patterns.md` 获取 8 种搜索方法的具体模式，包括文件名匹配、代码变量搜索、API 调用特征、配置文件、RAG 相关、模型参数、Few-shot 示例和工具 Schema。

**效率策略**：对于大型项目（超过 500 个源文件），用 Agent 子代理并行执行不同类别的搜索（如一个搜文件名模式，一个搜代码变量，一个搜 API 调用），然后合并结果。小项目直接顺序执行即可。

区分生产代码和测试代码（`test/`、`tests/`、`__tests__/`、`*_test.*`、`*.spec.*` 中的提示词标注为测试用途）——这个区分很重要，因为测试中的提示词是 mock 数据，不代表项目的实际 LLM 使用方式。

搜索完成后去重：同一提示词在定义处和调用处都出现时，以定义处为准，调用处标记为引用。

**第二阶段出口：提示词主清单（必须持久化到文件）**

搜索完成后，**必须**生成一份精确的"提示词主清单"，这是第三阶段的唯一输入。

**关键：必须将主清单持久化到文件**，不能只输出到对话中。大型项目执行过程中对话会被压缩，如果主清单只在对话中，压缩后将丢失，导致后续验证无法进行。

- ≤100 个提示词：写入单文件 `ai_analysis/translated_prompts/MANIFEST.md`
- \>100 个提示词：按功能模块拆分为 `ai_analysis/translated_prompts/MANIFEST_[模块名].md`（参见"大型项目执行策略"）

清单格式：

```markdown
# 提示词主清单

> 自动生成于第二阶段搜索完成后，用于第三阶段验证比对。

## [分类1] (X 个)
| # | 源文件路径 | 变量名/提示词名 | 简述 | 翻译状态 |
|---|-----------|---------------|------|---------|
| 1 | `path/to/file.md` | VARIABLE_NAME | 一句话描述 | 待翻译 |
| 2 | ... | ... | ... | 待翻译 |

## [分类2] (Y 个)
| # | 源文件路径 | 变量名/提示词名 | 简述 | 翻译状态 |
|---|-----------|---------------|------|---------|
| 3 | ... | ... | ... | 待翻译 |

---
总计: N 个（其中中文原文 M 个）
```

这份清单必须**逐项列出每一个**待翻译的提示词，不得使用"等"、"其他 X 个"等省略表述。第三阶段的验证将以此清单为基准——如果清单遗漏了某个提示词，后续阶段也会遗漏。因此在生成清单后，应快速回顾各搜索来源，确认没有遗漏。

特别注意容易遗漏的提示词来源：
- 配对的 system/user 提示词（如有 `xxx_system.md` 通常还有 `xxx_user.md`）
- 中英文双语变体（如 `xxx.md` + `xxx_en.md`）
- 代码中内联的硬编码提示词（如 `sys_prompt = """..."""`）
- Agent 模板 JSON 文件中嵌套的 `sys_prompt` 字段
- `_r` 后缀的推理模型变体（内容可能与原版相同但需要独立文档）

### 第三阶段：提示词文档化

读取 `templates/doc_template.md` 获取翻译文档和索引的模板。

**核心要求：必须为每个提示词创建独立的翻译文档文件**，不能只生成索引。每份翻译文档包含原文、中文翻译、关键参数、提示词链路等完整信息（详见模板）。全部翻译文档创建完成后，再生成 `INDEX.md` 索引指向这些文件。

之所以保留英文原文是因为翻译可能丢失细微语义，原文方便校对。上下游链路记录是为了在报告中绘制提示词串联关系。

**大量提示词的并行策略**：当提示词数量超过 10 个时，**必须**按功能模块分组，使用多个 Agent 子代理并行生成翻译文档。每个子代理负责一个模块（**5-10 个**提示词，不超过 15 个）。每个子代理的 prompt 中必须包含：
1. 要翻译的提示词完整列表（含文件路径、行号、变量名）
2. 输出目录：`ai_analysis/translated_prompts/`
3. 翻译模板文件路径：让子代理自行读取模板文件（`[skill目录]/templates/doc_template.md`），不要在 prompt 中内联模板内容——这样可以节省主 agent 构建 prompt 时的上下文消耗
4. 明确指令：子代理需用 Read 工具自行读取每个提示词的源文件获取原文内容（不要在 prompt 中内联所有原文，避免超长），然后为**每一个**提示词创建一个独立的 `.md` 文件，用 Write 工具逐个写入

**波次调度**：并行子代理数量不超过 6 个。如果提示词分组超过 6 组，按波次派发：
- 第一波：派发前 6 组（或更少），等待全部返回
- 验证第一波的文件实际写入（Glob 检查）
- 第二波：派发剩余分组
- 重复直到所有分组完成
- 每波之间更新 MANIFEST.md 中的翻译状态列

#### 子代理失败处理（关键）

子代理可能因权限拒绝、超时或其他原因失败。**必须**执行以下分级重试策略：

**第一级：重新派发子代理（优先）**
1. 每个子代理完成后，检查其返回状态——如果子代理报告了错误或被拒绝权限，**立即标记该批次为未完成**
2. 分析失败原因，调整子代理的 prompt 后重新派发：
   - 权限拒绝：在新子代理 prompt 中预先说明需要 Read/Write 权限，或将批次拆分为更小的单元（每个子代理不超过 5 个文件）
   - 超时：将批次拆分为更小的单元重新派发
   - 内容错误：在 prompt 中补充更明确的指令和示例
3. 同一批次最多重试 **2 次**子代理

**第二级：主 agent 兜底（仅在子代理反复失败时）**
4. 如果同一批次连续 2 次子代理失败，才由主 agent 直接使用 Write 工具补写该批次的翻译文档
5. 主 agent 兜底时，应将源文件内容以最精简方式读入（只读提示词正文，跳过周围代码），避免消耗过多上下文

**通用规则**
6. 不得在任何子代理仍在运行时向用户报告"完成"——必须等待所有子代理（包括后台运行的）返回后才能进入验证步骤
7. 每次子代理返回后，用 Glob 检查该批次的文件是否实际写入磁盘，不能仅凭子代理声称"已完成"

#### 强制验证门控（第三阶段出口条件）

**在宣布第三阶段完成或进入第四阶段之前，必须执行以下验证流程。不满足条件时禁止继续。**

**步骤 1：建立源文件清单**
读取 MANIFEST 文件（单文件 `MANIFEST.md` 或多文件 `MANIFEST_*.md`），获取完整的源文件列表。如果 MANIFEST 因故不存在，用 Glob/Bash 重新扫描源文件重建。

**步骤 2：建立已生成清单**
用 Glob 扫描 `ai_analysis/translated_prompts/` 下所有 `.md` 文件（排除 INDEX.md 和 MANIFEST.md），生成"已翻译清单"。

**步骤 3：逐一比对映射**
将每个源文件与已翻译文件进行一对一映射匹配。**不能仅靠文件数量比较**——必须确认每个源文件都有对应的翻译文档。可以委派给一个 Explore 子代理执行比对（避免主 agent 消耗大量上下文来处理长列表）。输出一个差异表：
- 已覆盖：源文件名 → 翻译文件名
- 未覆盖：源文件名 → [缺失]
- 多余/重复：翻译文件名 → [无对应源文件或与另一文件重复]

**步骤 4：补缺**
如果存在未覆盖的源文件，**委派子代理**补写翻译文档（遵循子代理失败处理策略）。补缺完成后，**重新执行步骤 2 和步骤 3**确认全部覆盖。

**步骤 5：生成 INDEX.md**
只有在步骤 3 确认 0 个未覆盖时，才生成 INDEX.md 索引。INDEX.md 中的文件数量必须与实际 `.md` 文件数量一致。同时更新 MANIFEST.md 中所有条目的翻译状态为"已完成"。

**步骤 6：向用户报告**
报告时必须包含：总源文件数、已翻译数、覆盖率百分比。如有重复文件需说明原因。

### 第四阶段：项目分析报告

读取 `templates/report_template.md` 获取 8 章报告模板。生成 `ai_analysis/AI_MODEL_USAGE_ANALYSIS.md`，覆盖项目概述、架构数据流图、提示词统计、场景分析、工具清单、模型参数汇总、上下文工程分析、关键发现总结。

### 第五阶段：产品说明书

读取 `templates/guide_template.md` 获取 12 章说明书模板。生成 `ai_analysis/PRODUCT_GUIDE.md`。

这份说明书的目标读者是没有编程经验的人，所以避免直接使用技术术语——必须用时紧跟括号解释。写作时站在"想用这个项目但不知道从哪开始"的用户视角。

## 输出目录

```
ai_analysis/
├── translated_prompts/
│   ├── MANIFEST.md              ← 提示词主清单（≤100个时用单文件）
│   ├── MANIFEST_[模块名].md     ← 按模块拆分的主清单（>100个时使用）
│   ├── INDEX.md                 ← 翻译文档索引（第三阶段验证通过后生成）
│   └── [各翻译文档]
├── AI_MODEL_USAGE_ANALYSIS.md
├── PRODUCT_GUIDE.md
└── ERRORS.md                    ← 执行异常日志（有异常时生成，无异常则不创建）
```

## 执行规范

### 搜索策略
1. 优先使用 Glob 和 Grep 进行全局扫描，避免遗漏
2. 对于大型项目，按第一阶段的规模评估策略分目录逐步深入
3. 记录所有已搜索的路径和模式，确保覆盖完整
4. 每种搜索方法至少执行一次，即使前几种已经找到很多结果——不同方法覆盖不同类型的提示词

### 文件读取原则
1. 发现可疑文件后，完整读取相关内容
2. 对于代码文件，读取提示词所在函数或类的完整上下文（不限定固定行数，以理解完整逻辑为准）
3. 追踪导入和引用关系——一个提示词可能在 A 文件定义、B 文件组装、C 文件调用

### 按规模分级的执行策略

根据第二阶段识别出的提示词数量，选择对应的执行策略。规模越大，主 agent 越要"放手"，把工作下放给子代理和文件系统。

#### 小型（≤30 个提示词）

主 agent 可以直接操作：
- 搜索阶段可以 Read 源文件内容来确认提示词
- 翻译阶段可以分 2-3 个子代理并行完成，也可以主 agent 直接写
- 验证阶段主 agent 自己比对文件列表即可
- 报告阶段可以主 agent 直接写，也可以委派子代理

#### 中型（31-100 个提示词）

主 agent 做调度和验证，子代理做内容生产：
- **主 agent 不读取源文件内容**——所有源文件阅读和翻译文档写入都在子代理中完成
- 主 agent 只负责：(a) 分配任务给子代理，(b) 用 Glob 验证子代理输出的文件是否存在
- 子代理 prompt 中给出翻译模板文件路径，让子代理自己读模板、读源文件、翻译、写文件，一站式完成
- 按功能模块分组，每个子代理处理 **5-10 个**提示词，并行不超过 6 个，分波调度
- 验证阶段的映射比对可委派给一个 Explore 子代理完成
- 报告和说明书委派给独立子代理生成

#### 大型（101-300 个提示词）

在中型策略基础上，增加**模块级子协调器**：
- **MANIFEST.md 按模块拆分**：不再用单一 MANIFEST.md，而是为每个功能模块创建独立清单文件 `MANIFEST_[模块名].md`，每个文件只包含该模块的提示词列表（控制在 30 条以内）
- **模块协调器**：为每个模块派发一个"协调器子代理"，该子代理负责：
  1. 读取本模块的 `MANIFEST_[模块名].md`
  2. 按 5-10 个一批拆分并（在子代理内部）逐批翻译
  3. 完成后用 Glob 自验证本模块的文件齐全性
  4. 返回简短状态报告：`模块X: 已完成 M/N 个，缺失: [列表或"无"]`
- 主 agent 只需：(a) 生成各模块的 MANIFEST 文件，(b) 并行派发模块协调器（不超过 6 个），(c) 收集协调器状态，(d) 对报告"缺失"的模块重新派发
- 验证阶段：派发一个 Explore 子代理读取所有 MANIFEST 文件并与磁盘文件交叉比对

#### 超大型（300+ 个提示词）

在大型策略基础上：
- 模块协调器自身也分波执行（每波不超过 4 个协调器，确保稳定性）
- INDEX.md 改为按模块生成独立索引文件（`INDEX_[模块名].md`），最后由一个子代理合并为总索引
- 报告分章节由不同子代理并行生成，最后由一个子代理合并
- 主 agent 全程只做最顶层调度，不接触任何提示词内容

#### 通用规则（所有规模适用）

**上下文节约的具体做法**：
- 主 agent 搜索阶段只记录文件路径列表，不 Read 文件内容。需要了解内容的，委派 Explore 子代理，子代理返回一句话摘要
- 子代理 prompt 中给出模板文件路径让子代理自行 Read，不要在主 agent 中内联模板内容
- 验证阶段只比对文件名列表，不读取翻译文档内容
- 向用户汇报时使用简洁的统计数字和表格，不要逐个列出所有文件的详情

**禁止的做法**：
- 禁止在主 agent 中 Read 所有提示词源文件的完整内容——这是上下文溢出的最常见原因
- 禁止在主 agent 的 prompt 中内联超过 3 个提示词的完整原文给子代理——让子代理自己 Read
- 禁止一次性派发超过 6 个并行子代理——难以追踪状态且容易遗漏失败的代理

### 断点续传（上下文压缩后的恢复）

大型项目执行中，对话上下文**一定会**被压缩。skill 通过以下文件实现断点续传，即使上下文丢失也能恢复进度：

| 持久化文件 | 写入时机 | 用途 |
|-----------|---------|------|
| `ai_analysis/translated_prompts/MANIFEST*.md` | 第二阶段结束 | 提示词主清单（≤100个用单文件，>100个按模块拆分） |
| `ai_analysis/translated_prompts/*.md` | 第三阶段进行中 | 翻译文档本身就是进度证明 |
| `ai_analysis/translated_prompts/INDEX.md` | 第三阶段验证通过后 | INDEX.md 的存在说明第三阶段已完成 |
| `ai_analysis/AI_MODEL_USAGE_ANALYSIS.md` | 第四阶段 | 报告文件 |
| `ai_analysis/PRODUCT_GUIDE.md` | 第五阶段 | 说明书文件 |
| `ai_analysis/ERRORS.md` | 任何阶段（有异常时） | 异常日志，恢复时检查未解决的问题 |

**恢复逻辑**：如果在执行中发现 `ai_analysis/` 目录已存在部分产出文件，**不要从头开始**。检查哪些文件已存在，从未完成的阶段继续：
1. 如果 MANIFEST 文件存在（MANIFEST.md 或 MANIFEST_*.md） → 第二阶段已完成，跳过搜索，读取 MANIFEST 获取提示词清单
2. 如果 INDEX.md 存在 → 第三阶段已完成，跳过翻译
3. 如果翻译文档存在但 INDEX.md 不存在 → 第三阶段部分完成，从验证步骤开始（比对 MANIFEST 与已有翻译文件）
4. 如果 AI_MODEL_USAGE_ANALYSIS.md 存在 → 第四阶段已完成
5. 如果 PRODUCT_GUIDE.md 存在 → 第五阶段已完成

### 翻译质量要求
1. 保持技术术语的准确性（如 "chain of thought" 译为 "思维链"，不要意译为 "一步步思考"）
2. 保留原始格式（Markdown、JSON 结构等）
3. 占位符（如 `{variable}`、`{{template}}`）在翻译中保持原样
4. 对于有歧义的术语，添加译注

## 特殊情况

- **中文提示词**：标注位置，跳过翻译
- **加密/混淆内容**：标注 [无法解析] 并记录位置
- **动态生成的提示词**：追踪函数调用链和条件分支，还原模板，用 `{动态部分}` 标记可变内容
- **超大提示词**：分段翻译，保持结构完整
- **外部托管**：代码中只有 `prompt_id` 引用时，标注 [内容托管于外部平台: 平台名称]
- **不确定的内容**：标注 [待确认] 并说明原因

## 故障处理与诚实报告

**核心原则：宁可如实报告失败，也不虚报成功。**

详细的故障分类表、健康检查流程、禁止行为清单和降级交付规则，参见 [reference/fault_handling.md](reference/fault_handling.md)。

执行时必须读取该文件并遵守其中所有规则。关键要点：
- 所有异常记录到 `ai_analysis/ERRORS.md`
- 子代理失败：重试 2 次 → 主 agent 兜底 → 记录
- 每步操作后用 Glob 做健康检查
- 禁止吞错误、禁止用成功措辞包装失败、禁止跳过验证
- 允许降级交付，但必须如实披露未完成项

## 阶段门控与完成性约束

**这是整个 skill 最重要的规则。违反以下任何一条都会导致用户收到不完整的交付物。**

### 禁止虚假完成声明

1. **绝对禁止在未经验证的情况下声称任何阶段"已完成"、"全部搞定"、"翻译齐全"等**。只有执行了对应阶段的验证流程并确认通过后，才能使用"完成"类措辞。
2. **不得用模糊数量掩盖缺口**。报告数量时必须用精确数字（如"已创建 38 个翻译文档，源文件共 72 个，缺失 34 个"），禁止使用"60+"、"大部分"、"基本完成"等模糊表述。
3. **子代理的报告不等于事实**。子代理声称创建了 N 个文件，必须用 Glob 实际验证磁盘上存在 N 个文件后才能采信。子代理可能因权限拒绝而静默失败。
4. **ERRORS.md 不为空时，不得声称"全部完成"**。必须在报告中注明存在异常，并引导用户查看 ERRORS.md。

### 阶段间门控

每个阶段必须满足出口条件后才能进入下一阶段：

| 阶段 | 出口条件 | 验证方法 |
|------|---------|---------|
| 第一阶段 | 项目根目录已确认，LLM依赖已识别 | 输出项目概况摘要 |
| 第二阶段 | 所有 8 种搜索方法已执行，提示词清单已生成 | 输出精确的提示词总数和分类计数 |
| 第三阶段 | **每个**源文件都有对应翻译文档（或标记为[未翻译]），INDEX.md 已生成 | 执行"强制验证门控"6步流程 |
| 第四阶段 | 报告文件已写入磁盘 | 用 Glob 确认文件存在，用 Read 检查非空 |
| 第五阶段 | 说明书文件已写入磁盘 | 用 Glob 确认文件存在，用 Read 检查非空 |

**门控通过不等于零缺陷**：阶段可以在有少量已知失败的情况下通过门控（前提是失败已记录且已尝试修复），但不能在有未检测的失败时通过。区别在于"知道有 3 个没翻译完"（可以继续）vs"不知道有多少没翻译完"（不能继续）。

### 最终交付验证

所有五个阶段完成后，在向用户报告最终结果之前，必须执行一次全局验证：

```
验证清单（必须逐项执行并在报告中列出结果）：
□ ai_analysis/translated_prompts/ 下文件数量 = [精确数字]
□ 源提示词文件数量 = [精确数字]
□ 覆盖率 = [百分比]（目标：100%，如有缺失必须列出具体文件和原因）
□ INDEX.md 存在且条目数与翻译文件数一致
□ AI_MODEL_USAGE_ANALYSIS.md 存在且非空
□ PRODUCT_GUIDE.md 存在且非空
□ ERRORS.md 状态 = [无异常 / 有 N 个未解决异常]
```

如果验证发现问题，按以下优先级处理：
1. **可修复的** → 立即修复后重新验证
2. **不可修复但已记录的** → 在最终报告中如实披露，允许降级交付
3. **不可修复且未记录的** → 先记录到 ERRORS.md，再按第 2 条处理

## 交互与进度

1. 每完成一个阶段，简要汇报发现数量和关键信息——**使用精确数字，不使用模糊量词**
2. 发现不确定的内容时，标注 [待确认] 继续推进，不要中断等待确认
3. 完成后提供完整的文件清单和核心发现摘要
4. 如果某个阶段搜索结果为空（如没有工具定义），在报告中如实说明，不要编造
5. **如果中途发现之前阶段的产出有遗漏，必须立即回头补齐后再继续**——不要带着已知缺口向前推进
6. **如果出现无法解决的故障，立即如实告知用户**，说明：(a) 出了什么问题，(b) 已经尝试了什么，(c) 建议用户怎么做。不要自行决定降低交付标准而不通知用户
